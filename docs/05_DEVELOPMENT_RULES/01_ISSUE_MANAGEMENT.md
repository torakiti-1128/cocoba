# 01_ISSUE_MANAGEMENT (課題・タスク管理規約)

## 1. 基本方針
本プロジェクトにおけるIssue（チケット）は、単なる備忘録ではなく「AIと自分自身への厳密な作業指示書」として機能させる。Issue化されていない作業（思いつきのコード変更や場当たり的な修正）は、いかなる場合も`dev`および`main`ブランチへのマージを認めない。

## 2. タスクの粒度
Issueのスコープが大きすぎると、AIは文脈を見失ってハルシネーションを起こし、人間はコードレビュー不能に陥る。以下の基準でタスクを徹底的に細分化する。

* **絶対ルール:** 1つのIssueは「半日〜最大1日」で完了・テスト検証できる最小単位まで分解すること。
* ❌ 悪手: 「AI推論システムを構築する」（巨大すぎる）
* ✅ 正手: 「YOLOv8モデルをN100向けにONNX形式へ変換するスクリプトの作成」
* ✅ 正手: 「エッジPCでのRTSPストリーム受信とフレームドロップ処理の実装」

## 3. Issueの記載フォーマット
新規にIssueを作成する際は、必ず以下の3要素を明記する。これがそのままAIに作業を丸投げする際の「プロンプトの前提条件」となる。

1. **背景・目的:** なぜこの作業が必要か（例：推論速度の向上、メモリリークの防止）。
2. **完了条件 (Acceptance Criteria):** 何を満たせばこのIssueをクローズできるか。可能な限り定量的な基準を設ける（例：ONNX変換エラーが出ないこと、推論速度が10FPSを維持できること）。
3. **参照ドキュメント:** このタスクを実行する上で絶対に従うべき設計図のファイルパス（例：`docs/03_INTERNAL_DESIGN/01_VISON_ALGORITHM.md` を参照せよ）。

## 4. ステータスとライフサイクル
GitHubのカンバンボードを用い、Issueを以下のステータスで厳格に管理する。

* **未着手:** 着手を待っている状態。
* **作業中:** ブランチが切られ、進行中の状態。
* **完了:** `dev`（または`main`）にマージされ、完全にクローズされた状態。

## 5. Issueとブランチ・PRの紐付け
* **1 Issue = 1 Branch = 1 PR** の原則を厳守する。複数のIssueを1つのブランチで同時に解決しようとしてはならない。
* ブランチ作成時は、Issueの内容に基づく命名を行う（例: Issue #12 がカメラ連携なら `feat/camera-rtsp`）。
* PRのタイトルおよび説明文には、必ず `Closes #Issue番号` （例: `Closes #12`）と記載し、PRマージ時に該当Issueが自動でクローズされるGitHubの連動機能を活用する。
