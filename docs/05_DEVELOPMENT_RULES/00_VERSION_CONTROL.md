# 00_VERSION_CONTROL (バージョン管理・Git運用規約)

## 1. 基本方針
本プロジェクトはIoT・ハードウェアを含む実稼働システムであるため、物理的な事故を防ぐための厳格なブランチ戦略（Git Flowベース）を適用する。

* **`main` ブランチ:** 結合テストおよび実環境テストをパスし、「いつ実機に書き込んでも安全に稼働する」コードと、承認済みのドキュメントのみを配置する。直接のコミットは禁止。
* **`dev` ブランチ:** 開発の主軸。実装工程における機能追加やバグ修正は、すべてこのブランチに向けて統合される。

## 2. ブランチ運用と命名規則
作業を行う際は、必ず `dev`（ドキュメント策定フェーズであれば `main`）から派生ブランチを作成する。ブランチ名は「すべて小文字のケバブケース（ハイフン区切り）」とする。

* `docs/xxx-xxx` : ドキュメントの作成・修正 (例: `docs/00-overview`)
* `feat/xxx-xxx` : 新機能の開発、モジュールの追加 (例: `feat/yolo-inference`)
* `fix/xxx-xxx` : バグ修正、ハードウェアの不具合対応 (例: `fix/motor-torque`)
* `test/xxx-xxx` : テストコードの追加、検証作業 (例: `test/camera-fps`)
* `hw/xxx-xxx` : ハードウェア・3Dモデル・回路図の変更 (例: `hw/dispenser-cad`)

## 3. コミットメッセージ規約
AIが進捗を正確にトラッキングし、かつタスクのトレーサビリティを確保するため、[Conventional Commits](https://www.conventionalcommits.org/) に準拠し、**必ず末尾に対応するIssue番号を付与する**ことを強制する。

**【フォーマット】**
`<type>(<scope>): <subject> (#<Issue番号>)`

**【Type一覧】**
* `feat`: 新機能（Pythonコード、ESP32実装など）
* `fix`: バグ修正
* `docs`: ドキュメントのみの変更
* `style`: コードの動作に影響しないフォーマット修正（Linter対応など）
* `refactor`: バグ修正や機能追加を行わないコードの書き換え
* `test`: テストの追加・修正
* `chore`: ビルドプロセスやツール、インフラ（Docker等）の変更

**【良いコミット例】**
* ✅ `feat(vision): YOLOv8による排泄物のバウンディングボックス抽出を実装 (#12)`
* ✅ `fix(esp32): Wi-Fi切断時の再接続ループによるウォッチドッグタイムアウトを修正 (#34)`
* ✅ `docs(rules): バージョン管理規約にdevブランチとIssue紐付けのルールを追加 (#5)`

**【禁止事項】**
* ❌ Issue番号の記載漏れ（追跡不可能な「孤児コミット」を生むため厳禁）。
* ❌ 複数のIssueにまたがる変更を1つのコミットにまとめること。

## 4. プルリクエスト (PR) 規約
PRは、AIが生成したコードをシステム（`dev` または `main`）へ統合するための「検疫所」である。以下の規約を満たさないPRは、いかなる理由があってもマージを却下する。

### 4.1. 粒度とIssueの自動クローズ
* **1 PR = 1 Issue** の原則を厳守する。
* PRの本文（Description）の冒頭には、必ず `Closes #Issue番号` を記載し、マージ完了と同時に該当Issueが自動でクローズされるGitHubの連動機能を活用すること。

### 4.2. PRテンプレートの強制
PRを作成する際は、必ず以下のフォーマットを本文に記述し、チェックリストを埋める義務を負う。

```markdown
Closes #【Issue番号】

### 概要
* 【このPRで何を解決したのか、アーキテクチャにどう影響するのかを端的に記載】

### 検証・テスト結果
* 【TDDに基づくテストの実行結果、またはハードウェアの動作ログ・スクショを添付】

### 品質・安全基準チェックリスト
- [ ] `02_CODING_RULE` に基づき、Linter/Formatterの警告がゼロである。
- [ ] （ハードウェア制御を含む場合）`03_HARDWARE_SAFETY` に反する物理的危険性（遅延、過電流リスク等）を排除した。
- [ ] AI起因の仕様の捏造（ハルシネーション）がないことを確認した。
```

### 4.3. マージ前のAIセルフレビュー
PRをマージする前には、実装を担当したAIとは別のペルソナ（QAのスペシャリスト）を呼び出し、差分に対する厳しいコードレビューを通過させる。

## 5. マージ戦略
1. **feature -> dev:** 作業ブランチでの実装・単体テスト完了後、`dev` ブランチに対してPRを作成する。マージ時は **「Squash and merge」** を原則とし、細かいコミット履歴を1つに圧縮して `dev` をクリーンに保つこと。
2. **dev -> main:** `dev` ブランチ上でシステム全体の統合テスト（シナリオテスト）が完了し、リリース可能な品質に達した段階で `main` へマージする。マージコミットを作成し、リリースタグ（例: `v1.0.0`）を打つこと。
