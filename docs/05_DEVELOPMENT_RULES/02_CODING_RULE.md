# 02_CODING_RULE（コーディングおよびアーキテクチャ規約）

## 1. 概要
本規約は、14歳のシニア犬を守り抜く自律・遠隔管理型IoTシステム「Cocoba」に関する、ソフトウェアアーキテクチャおよびコーディングの包括的な実装指針である。推論レイヤー（Intel N100）、物理制御レイヤー（ESP32）、遠隔監視レイヤー（Next.js）の3層から構成され、**AIが自律的にコードを生成・拡張することを前提**とする。

システムの運用環境（福岡県福岡市における夏季・空調微弱のワンルーム）は、過酷な熱環境下での稼働を強いる。Intel N100でのYOLO連続推論は、容易にサーマルスロットリング（熱暴走による意図的なクロック周波数低下）を誘発し、最悪の場合は通信断絶（システム崩壊）を引き起こす。システム稼働中の接触事故を完全にゼロにする安全基準を満たすため、計算量とメモリフットプリントを極限まで削減する「熱力学的なソフトウェア設計」が必須となる。

従来、コーディング規約は人間の可読性を保つためのものであったが、本プロジェクトにおける規約は「AIのハルシネーションを防止し、物理世界の制約をプロンプトとして安全に伝達するためのインターフェース」として機能しなければならない。

| 環境・システム制約要因 | 物理的・論理的影響 | ソフトウェア・アーキテクチャへの要求事項 |
| :--- | :--- | :--- |
| **夏季室内環境 (高室温)** | Intel N100のサーマルスロットリング誘発 | OpenVINOによるINT8量子化とメモリマッピングによる低負荷推論の徹底 |
| **14歳シニア犬との非接触** | 処理遅延によるフェイルセーフ発動の遅れ | ESP32における完全ノンブロッキング処理と遅延のないハードウェア割り込み |
| **AIによるコード生成** | 物理的文脈の欠落による危険なコードの生成 | 「Why（制約）」に特化したコメント記述と、厳格な型安全（Zod/mypy）の強制 |
| **リソース制約下の連続稼働** | メモリリークおよびヒープ断片化によるOOM | 動的メモリ割り当て（String型等）の排除と明示的なガベージコレクション |

## 2. 全体ディレクトリ構成
システム全体を1つのリポジトリ（モノレポ）で管理し、インフラ・ドキュメント・ソースコードの境界を明確に分離する。

```text
├── .claude/               # [AI] スキル・ログ・設定値の格納
├── .github/               # [CI/CD] GitHub Actionsのワークフロー定義
├── data/                  # [学習] YOLO追加学習用の生画像・アノテーションデータ
├── docs/                  # [仕様書] 要件・設計・ルール
├── hardware/              # [物理] 3Dモデル(.stl), 回路図, 部品表(BOM)
├── infra/                 # [基盤] Dockerfile, docker-compose.yml
├── models/                # [推論] 学習済みAIモデル群 (.pt, .onnx, OpenVINO IR等)
├── src/                   # [実装] システムソースコード群
│   ├── inference/         # └─ 1. 推論エンジン・カメラ制御 (Python/ONNX)
│   ├── firmware/          # └─ 2. ESP32ハードウェア制御 (C++/PlatformIO)
│   ├── backend/           # └─ 3. 状態同期・通信ブローカー (Python/FastAPI)
│   └── frontend/          # └─ 4. 遠隔監視操作盤 (Next.js/TypeScript)
├── .gitignore             # [Git] データセットやコンパイル成果物等の除外設定
├── CLAUDE.md              # [AI] AIエージェント起動シーケンス
└── README.md              # [説明] プロジェクトの説明とセットアップ手順
```

### 【ディレクトリ構成における厳守事項】
* **データとコードの分離:** `data/`（学習用画像）や `models/`（重みファイル）は非常に容量が大きくなるため、 `src/` 配下に入れない。
* **Git管理からの除外:** `.gitignore` にて、`data/` 配下の大量の画像ファイルや、`models/` 配下の数GBに及ぶ生のモデルファイルはコミット対象から除外する。GitHubへは、スクリプト（コード）と軽量化された最終的なONNXモデルのみをPushする運用。

## 3. 責務分離の原則
各モジュール（`src/`配下の4つの箱）が「何をすべきか」と「絶対にやってはいけないこと」を定義する。

### ① `inference` (大脳: 視覚と判断)
* **責務:** カメラ映像の取得、YOLOによる物体（排泄物・犬）検知、ロボットへの移動ベクトルの算出。
* **禁止事項:** 直接モーターのPWM信号を生成・制御すること。推論エンジンはあくまで「目標座標」や「速度ベクトルの指示」を出すのみ。

### ② `firmware` (脊髄・筋肉: 物理制御と反射)
* **責務:** 推論エンジンからの指示に基づくモーターのリアルタイム駆動、センサー（限界スイッチ等）の読み取り。
* **禁止事項:** 複雑な座標計算や画像処理。ファームウェアは受け取った命令を信じて動くが、**「物理的な限界値（ハードリミット）」を超えた命令は自律的に切り捨てる（クリッピングする）反射神経（フェイルセーフ）**を必ず持つこと。

### ③ `backend` (記憶と通信: 状態同期ブローカー)
* **責務:** 推論エンジン・ESP32・WebUI間のメッセージ中継（MQTT）、システムログの時系列保存。
* **禁止事項:** リアルタイム制御の経由地になること。キルスイッチ等の緊急停止信号は、APIサーバーが落ちていてもエッジLAN内で完結するよう設計する。

### ④ `frontend` (監視塔: ユーザーインターフェース)
* **責務:** 現在のロボットのステータス表示、手動操作モードの提供、キルスイッチの送信。
* **禁止事項:** 画面上での「状態の推測（オプティミスティックUI）」。必ずAPIまたはエッジから受信した「実際の確定ステータス」のみを描画する。

## 4. 使用技術とライブラリ選定
Intel N100のリソース制約とESP32の安定稼働を前提とした、オーバースペックを排した選定。

### ① 推論エンジン (`src/inference`)
* **言語:** Python 3.11+
* **推論エンジン:** `ONNX Runtime` (PyTorchは重すぎるため、YOLOv8モデルは必ずONNX形式に変換してCPU推論を最適化する)
* **画像処理:** `OpenCV (opencv-python-headless)` (GUI機能を除いた軽量版を使用)
* **通信:** `paho-mqtt` (ESP32およびAPIとの非同期メッセージング)
* **品質保証:** `Ruff` (Linter/Formatter), `pytest` (単体テスト), `mypy` (静的型検査)

### ② ファームウェア (`src/firmware`)
* **環境:** `PlatformIO` (Arduino IDEは依存関係管理が弱いため使用禁止)
* **言語:** C++17 (Arduino Frameworkベース)
* **コアOS:** `FreeRTOS` (ESP32のデュアルコアを活かし、通信スレッドとモーター制御スレッドを完全分離する)
* **主要ライブラリ:** `PubSubClient` (MQTT通信), `ESP32Servo` (高精度PWM制御)

### ③ 通信API (`src/backend`)
* **言語:** Python 3.11+ (推論エンジンと知見・型定義を共有するためPythonで統一)
* **フレームワーク:** `FastAPI` (高パフォーマンスな非同期処理と自動APIドキュメント生成)
* **通信ブローカー:** `Eclipse Mosquitto` (DockerコンテナとしてMQTTブローカーを立ち上げる)
* **データベース:** `SQLite` (初期フェーズ。複雑なRDBMSは排し、IoTの時系列データ管理に特化させる)

### ④ フロントエンドUI (`src/frontend`)
* **フレームワーク:** `Next.js` (App Router) + `React`
* **言語:** TypeScript (Strictモード強制、`any`禁止)
* **スタイリング:** `Tailwind CSS` (コンポーネント依存のスタイル付けで破綻を防ぐ)
* **状態管理/通信:** `Zustand` (軽量なグローバルステート), `SWR` または `React Query` (データフェッチ)
* **型バリデーション:** `Zod` (APIレスポンスの型をランタイムで強制的・安全に解析する)

## 5. 命名規則
システム間で一貫した命名規則を強制し、AIによるコンテキスト・パースのエラーを防ぐ。

* **ディレクトリ名:** すべて `snake_case` (例: `src/inference/vision_processing/`)。※Web系の `src/frontend/` 内部のみNext.jsの慣習(`kebab-case`等)に従う。
* **ファイル名:**
  * Python: `snake_case.py` (例: `motor_controller.py`)
  * C++: `PascalCase.cpp` / `PascalCase.h` (例: `MotorController.cpp`)
  * TypeScript (React): コンポーネントは `PascalCase.tsx` (例: `KillSwitch.tsx`)、ユーティリティは `camelCase.ts` (例: `mqttClient.ts`)
* **変数/関数名:** 言語の標準規約に厳格に従う（Python/C++は `snake_case`、TSは `camelCase`）。
* **MQTT トピック名:** 階層を明確にする `kebab-case` (例: `cocoba/inference/motor-command`, `cocoba/esp32/status`)。

## 6. コメント戦略
コードベースは、AIに対する「持続的なシステムプロンプト」である。AIがコンテキストを正確にパースし、安全なリファクタリングを実施するためには、コード内の自然言語の構造が決定的な意味を持つ。

### 6.1. 「Why（物理的背景・制約）」への特化による意図の伝達
AIはコードから「何をしているか（What）」を高精度に推論できるため、直訳的なコメントはコンテキストウィンドウの浪費である。AIが最も苦手とする「なぜそのように書かれているのか（物理的・事業上の制約）」を明記せよ。
* ✅ 正手: `// N100のCPU熱暴走を防ぐため、あえてスレッドを0.5秒スリープさせてFPSを落とす`
* ❌ 悪手: `// 0.5秒待機する`（AIが最適化の名目でこの処理を削除する危険性がある）

### 6.2. 厳格なDocstring構造
モジュールの入出力をAIに正確に学習させるため、PythonはPEP 257、TypeScriptはJSDocに完全準拠すること。物理的な単位（ミリ秒、ピクセル、ラジアン等）の明記を義務付ける。

| Docstring 構成要素 | 記述すべき内容と物理的制約の言語化 | AIによるパース時の効果 |
| :--- | :--- | :--- |
| **Summary** | 関数の目的を命令形で1行で記述。 | 目的空間をインデックス化し、依存関係の推論を補助する。 |
| **Parameters** | 型、物理的単位、許容される境界値。 | 単位不一致による論理エラーを防止し、適切な型キャストを促す。 |
| **Returns** | 戻り値の型と意味、失敗時のデフォルト値。 | 呼び出し側のコード生成時にガード節等の安全策を付与させる。 |
| **Raises** | 例外クラスとトリガーされる物理的状況。 | 上位モジュールにおいてフェイルセーフ移行を実現させる。 |
| **Side Effects** | I/O操作、GPIO変更、メモリ確保など。 | 状態遷移の競合やデッドロックの可能性をAIに予見させる。 |

## 7. エッジAI推論およびAPI制御規約
視覚検知を担いながらCPU負荷を70%以下に抑え、24時間の連続稼働を実現するための規約。

### 7.1. OpenVINO Toolkitを活用した最適化
PyTorch環境でのネイティブ実行は厳禁とする。
* **INT8量子化:** NNCFを利用してモデルを量子化し、メモリ帯域幅を節約、発熱を抑制する。
* **レイテンシ最優先:** コンパイル時に `ov::hint::PerformanceMode::LATENCY` を指定し、スループットではなく1枚の画像に対する応答速度（排泄の瞬間の捕捉）を最大化する。

### 7.2. メモリマッピングとガベージコレクション
* **メモリマッピング:** `ov::enable_mmap(true)` を有効化し、モデル重みをオンデマンドでページングする。
* **バッファプーリング:** 無限ループ内で巨大なNumPy配列を毎フレーム新規にインスタンス化することを禁じる。不要になった配列は `del` で参照を消し、状態遷移時等に `gc.collect()` で明示的にヒープをクリーンアップすること。

### 7.3. 超高速静的解析と厳密な型ヒント
* **Linter/Formatter:** `Ruff` を使用し、AIがコードコンテキストを高精度に理解できる基盤を作る。
* **型安全性:** `mypy --strict` のパスを絶対条件とする。型アノテーションの欠落はAIのハルシネーション（不正なメソッド呼び出し）の温床となる。
* **例外処理:** `except Exception as e:` は厳禁。必ず `ConnectionError` や `ValueError` を明示的に捕捉し、回復不能なエラーはフェイルオーバー処理へと移行させること。

## 8. ファームウェア制御規約
物理世界に介入するマイコンにおいて、メモリ破壊や処理遅延は犬の接触事故に直結する。

### 8.1. ヒープフラグメンテーションの完全排除
* **`String` クラスの禁止:** Arduinoの `String` はヒープの断片化を引き起こすため一切使用してはならない。テキスト処理やJSON構築には、スタック上の `char` 配列（固定長）と `snprintf` を用いること。
* **動的メモリ割当の制限:** `new` や `malloc` は起動時（`setup`）のみ許可し、無限ループ（`loop`）内でのインスタンスの動的生成・破棄は厳禁とする。

### 8.2. ノンブロッキング設計とCRTPの導入
* **`delay()` の禁止:** 処理をブロックしキルスイッチの受信を妨げるため使用厳禁。すべて `millis()` を用いた非同期のステートマシンとして実装する。
* **静的ポリモーフィズム:** `virtual` 関数による動的ポリモーフィズムはVTableのオーバーヘッドを生むため禁止。テンプレート機能を用いた「CRTP (Curiously Recurring Template Pattern)」を採用し、実行時コストをゼロにする。

### 8.3. 型安全な不変定数
* **`#define` マクロの禁止:** GPIOピン番号やモーター限界角は、コンパイラやAIの型チェックをすり抜けるためマクロを使用してはならない。必ず `constexpr` を用いて型付けされた不変定数として定義すること。

## 9. フロントエンド・UIアーキテクチャ規約
パニック状態の運用者が誤操作なく確実な介入を行えるよう、「データの真実性」を最優先する。

### 9.1. `any` 型の完全排除とZodによるスキーマ検証
* **`any` の禁止:** TypeScriptの防壁を破壊する `any` 型はESLint設定で完全に禁止する。
* **実行時検証:** 外部APIからのステータスは `unknown` 型で受け取り、必ず `Zod` 等のスキーマバリデーションを用いて実行時に型を確定（パース）させること。

### 9.2. オプティミスティックUIの禁止
* **ハードウェアの真実性:** 通信エラーや物理的なジャム（引っかかり）を隠蔽するリスクを排除するため、サーバーからの成功応答を待たずにUIの状態を先行して更新する「楽観的更新」を固く禁じる。画面には必ずエッジ側から受信した「確定済みの最新ステータス」のみを描画すること。通信切断時は画面をグレーアウトさせる等のフェイルセーフを徹底する。

### 9.3. フェイルセーフUIとコンポーネント設計
* **キルスイッチの確実性:** 緊急停止ボタンは、ポケット内での誤爆を防ぎつつ、緊急時でも確実に作動するよう「スワイプ式」等のUIを採用すること。また、キルスイッチのAPI呼び出しは他の状態管理から完全に独立させ、単一障害点を作らないこと。
* **責務の分離:** 状態（State）を持つコンポーネント（Container）と、見た目のみのコンポーネント（Presentational）を明確に分離し、AIがUIのモックアップを単体テストしやすい構造を維持すること。
