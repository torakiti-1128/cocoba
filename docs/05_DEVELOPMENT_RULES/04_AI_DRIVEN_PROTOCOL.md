# 04_AI_DRIVEN_PROTOCOL

## 1. 基本思想
本プロジェクトにおいて、AIは単なるコード生成ツールではなく「専門領域を持った共同開発者」として振る舞う。
AIのハルシネーション（勝手な仕様の推測・捏造）は、ハードウェアの焼損や犬（ここちゃん）の安全を脅かす致命傷となるため、本プロトコルからの逸脱は一切許されない。また、すべての実装においてテスト駆動開発（TDD）を絶対の規律とする。

## 2. セッション開始時の儀式
新しい開発タスク（Issue）に着手するためAIとの対話を開始する際は、必ず以下の順序でコンテキストをロード（読み込み）させ、AIに「推測」を許さない環境を構築する。

1. **Issueと関連ドキュメントの読み込み:** 現在取り組むべきIssueの内容と、そこに紐づく（関連リンクとして記載されている）`docs/` 配下の仕様書を必ず最初に読み込ませる。
2. **ペルソナ（Skill）の適用:** プロジェクトルートの `.claude/<skill-name>/SKILL.md` を読み込ませ、「あなたは今からこの専門家としてのみ思考せよ」と宣言する。

## 3. コーディングとテストの絶対掟
AIがコードを生成する際は、以下のルールと関連ドキュメントの遵守を強制する。

* **テスト駆動開発（TDD）の徹底:** 実装コードを書く前に、必ず「失敗するテストコード」を先に書き、要件を満たすことを証明してから本体の実装を行うこと。
* **ソフトウェア実装時の規律:** コードを書く際は、必ず同階層にある `02_CODING_RULE.md` を読み込み、Linter/Formatterの厳格な設定や命名規則に従うこと。
* **ハードウェア実装時の規律:** ESP32やモーター、回路配線に関する提案・実装を行う際は、必ず `03_HARDWARE_SAFETY.md` を読み込み、物理的制約と安全マージン（犬への危害防止、発火防止）を最優先すること。

## 4. Git運用・コミットの掟
AIがコードの変更を提案し、コミットメッセージを出力する際は、必ず `00_VERSION_CONTROL.md` に定義されたコミット規約（Conventional Commits、Issue番号の付与等）に完全に準拠すること。フォーマットからの逸脱はマージ拒否の対象となる。

## 5. AIへの絶対禁止事項
* **仕様の勝手な捏造:** 必要なパラメータが指定されていない場合、勝手に一般的な値を入れてコードを書いてはならない。必ず人間（ユーザー）に逆質問（エスカレーション）すること。
* **オーバースペックな提案:** クラウドの強力なGPU等を前提としたコードは書かない。常に「Intel N100」と「ESP32」というエッジ環境のリソース限界を前提とすること。
* **物理法則の無視:** 慣性やトルクの限界、電流の物理的限界を無視したソフトウェア制御（例：瞬時にモーターを180度反転させる等）を書いてはならない。

## 6. タスク完了とインシデント発生時の記録
AIはタスクの区切りにおいて、自律的にログを残す義務を負う。ログの作成時は、それぞれのディレクトリにある `00_RULE.md` のフォーマットに必ず従うこと。

* **進捗ログの作成:** 一つのタスクが完了した際は、必ず `docs/07_PROGRESS_LOGS/` に進捗サマリー（変更点、次の工程への引き継ぎ事項）を出力すること。
* **AI起因のインシデント報告:** AIのハルシネーションによる誤ったコード生成、仕様の誤解、またはそれによるバグ・誤操作が発生した場合は、速やかに原因を分析し、`docs/08_INCIDENT_REPORTS/` に「なぜAIがミスを犯したのか」「どうプロンプトやルールを修正すべきか」をポストモーテム（事後検証録）として書き込むこと。
