# INCIDENT_REPORT_RULE (インシデント報告・再発防止規約)

## 1. 基本方針
`08_INCIDENT_REPORTS` 配下のファイルは、単なるバグのメモではない。AIのハルシネーション、ユーザーの指示ミス、または予期せぬハードウェアの挙動によってシステムが破綻（または破綻しかけた）際に、その根本原因を究明し、**プロジェクトの規律（プロトコルやルール）をアップデートして二度と再発させないための「ワクチン」**である。
AIは自身がエラーを引き起こした場合、言い訳を排し、冷徹に原因を分析して本フォーマットに従い報告する義務を負う。

## 2. ログ作成のタイミング
以下のいずれかの条件を満たした際、AIは自律的に、またはユーザーの指示によりインシデントレポートを出力しなければならない。

* AIが `04_AI_DRIVEN_PROTOCOL` やハードウェアの安全規約を無視した危険なコードを出力したとき。
* AIが勝手に仕様を捏造（ハルシネーション）し、手戻りが発生したとき。
* 実機テストにおいて、設計段階では予期していなかった物理的エラー（モーターのトルク不足、異常な発熱、カメラの著しい誤検知など）が発覚したとき。

## 3. ファイル命名規則
進捗ログと同様に時系列順に並べつつ、「インシデント（障害）」であることが一目でわかる厳格な命名規則とする。

**形式: `YYYYMMDDHHMM_incident_summary.md`**
* 例: `202603161040_incident_hallucination_pwm_value.md`
* 例: `202603171820_incident_motor_torque_shortage.md`

## 4. ログ記述フォーマット
AIがインシデントレポートを生成する際は、以下のMarkdownテンプレートに完全に準拠すること。項目の省略は一切許されない。

---
### 【AI向け出力テンプレート】
---

# インシデント報告: 【ここに障害のサマリーを1行で記載】

* **タイムスタンプ:** YYYY-MM-DD HH:MM
* **関連Issue:** #XXX
* **担当AI Skill:** 【例: 組込みC++専門家】

## 1. 発生した事象（インシデント内容）
* 【システムに何が起きたか、またはAIがどのような致命的な誤答をしたかを具体的に記載】
* 【例: AIがESP32のコード生成時、`03_HARDWARE_SAFETY` を無視して `delay()` 関数を使用したブロッキング処理を出力した。】

## 2. ユーザーの指示（プロンプト）とAIの誤った出力
* **ユーザーの指示:** 【どのような指示を出したか】
* **AIの出力（抜粋）:** 【どのコード・提案が問題だったか】

## 3. 根本原因の分析（Root Cause）
* 【なぜAIは間違えたのか、あるいはなぜシステムは停止したのか。表面的なバグではなく、仕組みの欠陥を分析する】
* 【例: ユーザーのプロンプト内に「`03_HARDWARE_SAFETY` を読め」という明示的な指示が漏れていたため、AIが一般的なC++の文脈で推測して回答してしまった（コンテキスト不足）。】

## 4. 解決策・修正したファイル
* `src/【ファイルパス】`: 【どのようにコードを修正して事象を解決したか】

## 5. 再発防止策（ルール・プロンプトのアップデート）
* 【同じミスを二度と起こさないため、どのドキュメント（プロトコルやテンプレート）をどう修正すべきか、具体的なアクションアイテムを記載】
* 【例: `05_PROMPT_TEMPLATES.md` のコーディング用テンプレートに、「必ず `03_HARDWARE_SAFETY.md` を厳守せよ」の一文をハードコードして追記するよう提案する。】
