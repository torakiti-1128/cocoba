#  アーキテクチャ構成

## 1. 方針
本システムは、リアルタイム性が求められる「AI検知・ロボット制御」と、非同期でよい「遠隔監視・通知」を切り離す。
* **エッジ自律稼働の原則:** インターネット接続が途絶した場合でも、エッジPC内のローカルネットワーク通信のみで「検知〜誘導〜投下」のコア機能が完結すること。Webシステムはあくまで「状態のミラーリングと介入」のみを担う。
* **低レイテンシの担保:** カメラ映像のAI推論やロボットの座標計算をクラウドにオフロードしない。すべて現場（エッジPC）で処理し、通信遅延による対象（犬）との接触事故を防ぐ。

## 2. システムレイヤー構成 (System Layers)
システムは大きく3つの層（Layer）に分割して構築する。
### Layer 1: エッジ・デバイス層 (現場環境)
物理的なアクションと即時判断を行うクローズドな環境。

* **エッジPC (司令塔):** AI推論（YOLO）、状態遷移（ステートマシン）の管理、各デバイスへイベント発行を行う。
* **視覚デバイス (カメラ):** 俯瞰映像をエッジPCへストリーミングする (RTSP)。
* **誘導デバイス (おやつシューター):** エッジPCからの指令で動作するマイコン制御デバイス。
* **実行デバイス (移動ロボット＆投下機構):** エッジPCから計算された座標を受け取り、移動・投下を実行するマイコン搭載ロボット。

### Layer 2: クラウド層 (遠隔管理)
状態の永続化と、外部からのアクセスポイントを提供する。

* **Web API / 状態同期サーバー:** エッジPCから定期的に送信されるステータス（待機、検知、出動中、完了、エラー等）を受信し、データベースを更新する。
* **通知ブローカー:** 状態の変化（特に投下完了や異常発生時）をトリガーに、LINE Messaging API等を経由して飼い主へプッシュ通知を送信する。

### Layer 3: クライアント層 (UI)
飼い主が状況を把握し、介入するための接点。

* **Webダッシュボード (スマホ/PC):** 現在のシステム状態、エッジPCからアップロードされた「排泄検知時のスナップショット画像」、ロボットの残弾数（ドームの数）を閲覧する。
* **緊急介入インターフェース (キルスイッチ):** 遠隔から「システムの即時停止」や「ロボットの強制帰還」のコマンドをWeb経由でエッジPCへ送信する。

## 3. データフローと通信プロトコル
各層間の通信は、用途に応じて最適なプロトコルを選定する。

1. **[エッジ内] 映像ストリーム:** カメラ →(RTSP)→ エッジPC
2. **[エッジ内] デバイス制御:** エッジPC →(HTTP/REST または ローカルMQTT)→ マイコン（誘導機・ロボット）
3. **[エッジ → クラウド] 状態同期:** エッジPC →(HTTPS POST / WebSocket)→ Webサーバー
   * 状態が変化したタイミング、または一定間隔（ハートビート）で送信。
4. **[クラウド → エッジ] 遠隔操作:** Webサーバー →(WebSocket / MQTT)→ エッジPC
   * ポーリング（定期的な問い合わせ）による遅延を防ぐため、双方向通信を確立し、キルスイッチの即時性を担保する。
