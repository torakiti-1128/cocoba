# 00_REQUIREMENTS_DEFINITION.md（要件定義書）

## 1. 業務要件 
本システムは、14歳のシニア犬（ここちゃん）の排泄物踏み散らかしを防止し、運用者（飼い主）の清掃コストと精神的負担を削減するための自律・遠隔管理型IoTシステムである。

* **BR-01:** 帰宅後の排泄物関連の清掃時間を「毎日3分以内」に短縮する。
* **BR-02:** システム稼働による犬への接触事故・怪我を「完全にゼロ」とする。
* **BR-03:** 外出先からのステータス確認と緊急介入を可能とし、運用者の不安を解消する。

## 2. 機能要件
システムが必ず実装しなければならない具体的な機能。これらが一つでも欠ければリリース不可とする。

### 2.1 視覚・検知機能
* **FR-11:** ネットワークカメラのストリーミング映像(RTSP)をエッジPCで常時受信できること。
* **FR-12:** AIが映像内から「犬」「排泄ポーズ」「排泄物」を識別できること。
* **FR-13:** 検知した「排泄物」および「ロボット上のArUcoマーカー」のカメラ画像上のピクセル座標を、床面の2D絶対座標(X, Y)へ変換できること。

### 2.2 誘導機能 (Decoy)
* **FR-21:** エッジPCからのAPIリクエストを受信し、ESP32マイコン経由でおやつシューターを稼働できること。
* **FR-22:** 1回の稼働につき、少量の誘引餌を正確に排出できること（過剰給餌の防止）。

### 2.3 ロボット制御・投下機能
* **FR-31:** エッジPCが計算した目標座標へ、ベースロボットを自律移動（ビジュアルサーボ制御）できること。
* **FR-32:** 目標座標到達後、投下機構を稼働させ、シールド（ドーム）を「1つだけ」落下させることができること。
* **FR-33:** 投下完了後、ロボットが初期待機位置（または充電ドック）へ自動で帰還できること。

### 2.4 遠隔管理・Web連携機能
* **FR-41:** エッジPCからWebサーバーへ、現在のステータス（待機中/処理中/エラー等）およびドーム残弾数を定期同期できること。
* **FR-42:** 排泄物検知時に、その瞬間のスナップショット画像をWebサーバーへアップロードできること。
* **FR-43:** 投下完了時、またはエラー発生時に、飼い主のスマートフォンへプッシュ通知を送信できること。
* **FR-44:** Webダッシュボードから「緊急停止（キルスイッチ）」コマンドを発行し、エッジPCへ即時反映できること。

## 3. 非機能要件
システムの品質、性能、安全性に関する制約。現場の実用性を担保する。

### 3.1 安全性・フェイルセーフ
* **NFR-11 (近接停止):** ロボット稼働中、カメラ映像上でロボットと犬の距離が「1.0m以内」に接近した場合、いかなる状態でもロボットを1秒以内に強制停止させること。
* **NFR-12 (タイムアウト):** 誘導餌の排出後、15秒経過しても犬が排泄物から1.5m以上離れない場合は出動をキャンセルし、待機状態へ戻ること。
* **NFR-13 (通信断絶):** エッジPCとロボット間の通信が1秒以上途絶えた場合、ロボット側マイコンのフェイルセーフによりモーター出力をゼロにすること。

### 3.2 性能・パフォーマンス
* **NFR-21 (推論速度):** エッジPC上でのYOLO推論は、CPU負荷を70%以下に維持しつつ、最低 10FPS 以上で実行できること。
* **NFR-22 (キルスイッチ応答):** Webダッシュボードから緊急停止ボタンを押下してから、エッジPCがコマンドを受信しロボットを停止させるまでの遅延は「2秒以内」とすること。
  * ※この低レイテンシを実現するため、WebUI (Vercel) は AWS IoT Core への MQTT over WebSocket 直接接続を行い、中継の API Gateway を介さずにコマンドを送信する。

### 3.3 運用性・保守性
* **NFR-31 (日次SOP):** 運用者の「出発前セットアップ」および「帰宅後リカバリ」の物理的作業が、それぞれ3分以内で完了するハードウェア設計であること。
* **NFR-32 (耐熱・連続稼働):** エッジPCは24時間365日の稼働を前提とし、サーマルスロットリング（熱暴走による性能低下）を起こさない排熱性能を持つこと。

## 4. 運用ユースケース

## 4.1. 運用の基本原則
* **ゼロ・タッチ・オペレーション:** 日中の無人環境下において、飼い主の能動的な介入（Web画面に張り付いて監視するなど）を一切不要とする。通知は「事後報告」または「異常発生時」のみに限定する。
* **物理タスクの極小化:** 出発前および帰宅後の物理的なセットアップ（弾込め、清掃）を簡単にするハードウェア設計・配置を前提とする。

## 4.2 運用ユースケース 
Webシステム連携を含めた、実運用における代表的なシナリオとシステムの振る舞い。

* **UC1: 正常稼働と事後確認**
  * **状況:** 留守中にここちゃんが排泄。
  * **システム:** AI検知 → おやつ誘導 → ロボット出動・投下 → 帰還を完全自律で完了。
  * **Web/通知:** 投下完了後、飼い主のスマホへ「シールド完了」のLINE通知と、検知時のスナップショット画像を送信。
  * **飼い主の行動:** 通知を見て安心し、そのまま業務・外出を継続する。

* **UC2: 遠隔キルスイッチの発動 (緊急介入)**
  * **状況:** AIが排泄物を検知してロボットが出動したが、スナップショット画像を見た飼い主が「ただの黒い靴下（誤検知）」であることに気づいた。あるいは、ロボットの挙動がおかしいことに気づいた。
  * **システム:** 稼働中ステータスをWebへ同期中。
  * **飼い主の行動:** Webダッシュボードを開き、即座に `[緊急停止]` をタップ。
  * **結果:** エッジPCがコマンドを即時受信し、ロボットのモーターへの電力供給を論理的にカット。誤動作による事故を未然に防ぐ。

* **UC3: 誘導失敗によるフェイルセーフ発動**
  * **状況:** 排泄後、ここちゃんがおやつに反応せず、排泄物のすぐ横で寝転がってしまった。
  * **システム:** エッジPCが「犬が安全圏に退避していない（タイムアウト）」と判断。ロボットの出動をキャンセル。
  * **Web/通知:** 飼い主へ「誘導失敗：出動をキャンセルしました」のアラートを送信。二次被害（踏み散らかし）のリスクがあることを事前に伝える。

## 4.3 業務フロー
飼い主が毎日必ず実行する物理的・論理的なアクション。このフローが守れない日は、システムを「稼働させない（電源OFF）」にすること。

### フェーズ1: 出発前
**【目標所要時間: 1分以内】**
1. **シールドの装填:** 洗浄・乾燥済みのドームを、Cocobaロボットのディスペンサーにスタック（最大積載数）する。
2. **誘引餌の補充:** おやつシューター（ESP32機）のサイロに、1日分のカリカリ（数粒×想定回数分）を投入する。
3. **ヘルスチェック (Web確認):** スマホでダッシュボードを開き、以下の「オールグリーン」を確認する。
   * エッジPCのステータスが `[待機中]` であること。
   * ロボットのドーム残弾数が正しく反映されていること。
   * カメラの最新映像に死角がないこと（カメラの向きズレ確認）。
4. **外出:** そのまま家を出る。

### フェーズ2: 日中
**【目標所要時間: 0分 (通知確認のみ)】**
1. システムからのPush通知（LINE等）を受信した際のみ、内容（スナップショット画像）を確認する。
2. 異常（誤検知や危険状態）を視認した場合のみ、Webダッシュボードから介入（キルスイッチ）を行う。

### フェーズ3: 帰宅後
**【目標所要時間: 3分以内】**
1. **システムの論理停止:** 帰宅時の飼い主の足などを誤検知しないよう、帰宅直後にWebダッシュボードから `[システムスリープ]` を実行し、監視を一時停止する。
2. **物理回収:** 床に配置されているドームを確認。ドームの上からトイレットペーパー等を被せ、中の排泄物ごと持ち上げてトイレ等へ廃棄する。
3. **清掃とリセット:**
   * 使用済みドームの裏面（フチ）をアルコール除菌シート等で拭き取る、または水洗いする。
   * Cocobaロボットを所定の待機位置（充電ケーブル接続）に戻す。

## 4.4 定期保守
システムの稼働率を100%に保つための、週末等を活用したメンテナンス作業。

* **[週次] センサー・カメラ清掃:** ネットワークカメラのレンズ、およびロボットに貼付したArUcoマーカーのホコリを拭き取る（認識率低下の防止）。
* **[週次] おやつシューター清掃:** カリカリの油分による「弾詰まり」を防ぐため、排出機構（筒やモーターの羽）を清掃する。
* **[月次] ロボットの足回り点検:** ルンバ（または自作ローバー）のタイヤやキャスターに、犬の毛やホコリが絡まっていないか確認・除去する。
* **[月次] エッジPC再起動:** メモリリークやOSの隠れた不具合をリセットするため、ミニPCを計画的に再起動（リブート）する。
  
## 5. 前提条件と制約事項
* **CON-01:** システムはワンルームでの運用を前提とする。
* **CON-02:** クラウドシステムがダウンした場合でも、エッジPC内のローカルネットワーク通信のみで「検知〜投下」のコア機能は完結すること（エッジ自律の原則）。
