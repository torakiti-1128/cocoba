# 06_NETWORK (通信プロトコル設計)

## 1. 基本方針
14歳のシニア犬に対する物理的接触事故を完全にゼロにするため、キルスイッチ（緊急停止）およびフェイルセーフ（異常時の自動停止）の即時性を最優先とする。システム全体のメッセージング基盤を**「MQTTプロトコル」**で統一する。HTTP(S)によるポーリングは、遅延と無駄なトラフィックを生むため状態同期・制御においては使用しない。

## 2. 通信レイヤー別 技術選定とアーキテクチャ

### ① [エッジ内] 映像ストリーム
* **経路:** カメラ → エッジPC (Intel N100)
* **プロトコル:** `RTSP`
* * **制約:** 映像はエッジPC内でのYOLO推論のみに使用し、原則としてクラウド（WebUI）へは生の映像ストリームを常時送信しない（帯域圧迫とN100のエンコード負荷による熱暴走を防ぐため）。WebUI側へは、検知時の座標や、低FPSの軽量なJPEGスナップショットのみを送信する。

### ② [エッジ内] ハードウェア制御
* **経路:** エッジPC (推論エンジン) ⇔ ESP32 (ファームウェア)
* **プロトコル:** `MQTT` (ローカルネットワーク内)
* **ブローカー:** エッジPC上で稼働するローカルの `Eclipse Mosquitto`
* **設計意図:** ESP32は複雑なHTTPスタックを持たず、PubSubClientを用いた非同期MQTT通信に特化させることで、メインループ（モーター制御）のブロッキングを完全に防ぐ。

### ③ [エッジ ⇔ クラウド] 状態同期・遠隔操作 (キルスイッチ)
* **経路:** エッジPC (APIブローカー) ⇔ クラウド上のMQTTブローカー ⇔ WebUI (Next.js) / FastAPI
* **プロトコル:** `MQTT over TLS` (エッジPC側) / `MQTT over WebSocket` (Next.js側)
* **設計意図:**
  1. **NAT/Firewall越え:** エッジPCからクラウドに対してアウトバウンドで接続を維持するため、一般家庭のルーター環境でもポート開放なしで双方向通信が成立する。
  2. **到達保証:** キルスイッチ等の重要コマンドは必ず `QoS 1`（少なくとも1回伝達）以上で送信し、パケットロスによる制御不能状態を防ぐ。
  3. **Last Will and Testament (LWT / 遺言機能):** エッジPCとクラウド間の接続が不意に切断された場合、クラウド側のブローカーが自動的に「通信途絶」トピックをNext.jsへパブリッシュし、UIを瞬時にロックアウト（操作不可状態に）して運用者に警告する。

## 3. WebAPI (REST / HTTP) の限定的な利用用途
状態のリアルタイム同期やロボット制御にHTTPは使用しない。HTTP(FastAPI)の用途は以下の「永続化された静的データの取得」のみに限定する。

* 過去の排泄検知ログ（時系列データ）の取得
* 過去のエラーログ、インシデントレポートの取得
* 初期設定情報（ロボットの動作パラメータのマスターデータ等）の取得

## 4. トピック設計規約 (Topic Naming Convention)
すべてのMQTTトピックは、以下の階層構造に従い、責務を明確に分離すること。

* `cocoba/edge/status/#`: エッジからクラウドへ送信される現在の状態（稼働中、検知状態、エラー等）
* `cocoba/edge/telemetry/#`: ESP32から吸い上げたセンサー値やモーター負荷情報
* `cocoba/cloud/command/#`: クラウドからエッジへ下される操作命令（手動移動、モード切替）
* `cocoba/system/kill-switch`: **【最優先】** システム全体を即時停止させる緊急停止トピック。このトピックを受信した場合、各モジュールは自律的に現在のアクションを破棄し、モーターへの通電を遮断すること。
