# ソフトウェア構成

## 1. 開発環境
環境構築の再現性と安定性を担保する。

* **エッジOS:** Ubuntu Linux (22.04 LTS または 24.04 LTS) 
* **言語 (エッジ):** Python 3.11+
* **言語 (デバイス):** C++ (Arduino Framework / ESP32向け)
* **Webシステム (クラウド):** 
  * フロントエンド: **Next.js 14 (App Router) / Vercel デプロイ**
  * バックエンド: **AWS Lambda (Python)**
  * データベース: **Amazon DynamoDB**
  * API エントリ: **Amazon API Gateway**

## 2. エッジAI・画像処理スタック 
Intel N100等のミニPCを利用し、熱負荷を抑えつつリアルタイム性を確保する。

* **物体検出モデル:** YOLOv8n (Nano)
* **推論エンジン:** ONNX Runtime
  * ※PyTorchの標準推論ではなく、必ず推論専用エンジンに変換（エクスポート）して実行し、CPU負荷を抑制する。
* **学習環境:** 開発PC (Local Training) / PyTorch, CUDA
* **空間認識・ナビゲーション:** OpenCV
  * ロボット天板のArUcoマーカーを検出し、ピクセル座標から床面の2D絶対座標(X, Y, 角度θ)へリアルタイムに変換する。

## 3. エッジ・クラウド間 通信プロトコル
現場の自律稼働を妨げず、かつ遠隔からの即時介入を可能にする非同期通信。

* **状態のアップロード (エッジ → Web):** `HTTPS POST` 
  * 宛先: **Amazon API Gateway**
  * エッジPCのステートマシンが遷移した瞬間にクラウドのDBを更新し、スナップショット画像を S3 へアップロードする。
* **遠隔介入・キルスイッチ (Web ⇔ エッジ):** `MQTT`
  * ブローカー: **AWS IoT Core**
  * 飼い主がWeb画面の「緊急停止」ボタンを押した際、IoT Core 経由でエッジPCへ即座に割り込みコマンドを送信する。

## 4. エッジ・デバイス間 通信プロトコル
LAN内で完結し、クラウドの障害に影響されないローカル通信。

* **ハブ:** **Eclipse Mosquitto (Local MQTT Broker)**
* **エッジPC ⇔ ESP32 (ロボット / シューター):** `Local MQTT`
  * 常に数十ミリ秒単位で「前進」「旋回」「停止」のコマンドを送り続ける必要があるため、オーバーヘッドの少ないプロトコルを選定する。
  * フェイルセーフ: ロボット側のESP32は、通信が「1秒以上途絶えた」場合、即時停止するウォッチドッグタイマーを実装する。

## 5. Webアプリケーション要件 (Vercel)
飼い主の不安を解消し、運用負担を下げるための必要最小限のUI。

1. **ステータスモニター:** 現在のシステム状態（待機中 / 処理中 / エラー）と、直近の検知画像。
2. **インベントリ管理:** ロボットに積載されている「ドームの残弾数」。投下ごとにカウントダウンし、ゼロの場合は警告を出す。
3. **マニュアル・オーバーライド機能:**
   * `[緊急停止 (Kill Switch)]`: 実行中の全シーケンスを破棄し、ロボットをその場で停止させる。
   * `[強制帰還 (Return to Base)]`: ロボットを初期位置へ戻す。
4. **プッシュ通知:** LINE Messaging API等を利用して飼い主のスマートフォンへ通知する。
